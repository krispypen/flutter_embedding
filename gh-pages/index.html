The Perfect Use Case: SDK Development

Build your SDK once in Flutter, distribute it to clients on any platform

Building a UI-Based SDK?

Imagine you're developing an SDK with rich UI components — a payment flow, an onboarding wizard, a chat interface, or any complex user experience. Your clients are diverse: some build native iOS apps, others use Android, React Native, or web frameworks like React and Angular.

Without Flutter Embedding CLI: You'd need to build and maintain separate codebases for each platform — native Swift/Kotlin, React Native bridges, web components. That's 5+ implementations to keep in sync.

With Flutter Embedding CLI: Build your SDK once in Flutter, then generate ready-to-distribute packages for every platform. Your clients integrate it like any other native library — they don't even need to know it's Flutter inside.

Why Flutter Embedding CLI?

Stop wrestling with build configurations. Start shipping features.

The Problem with Add-to-App

Flutter's official "add-to-app" approach is notoriously complex and painful to set up. It requires:

  • Manual configuration of build systems
  • Managing framework dependencies across platforms
  • Writing boilerplate communication code
  • Error-prone and time-consuming setup

The Solution

Flutter Embedding CLI eliminates this pain. With a single command, it generates production-ready modules with:

  • Pre-configured build setup and dependency management
  • Type-safe communication using Protocol Buffers
  • Ready-to-use example apps for immediate testing
  • Proper packaging (CocoaPods, AAR, npm)

Features

Define your communication contract in Protocol Buffers — the possibilities are endless

Define Your Own API

Use Protocol Buffers to define your exact communication contract. Theme switching, user data, complex objects — the possibilities are endless.

Bidirectional Communication

Define services for Flutter-to-host and host-to-Flutter calls. Both directions are fully type-safe with generated code.

Auto-Generated Code

Write your proto once, get type-safe clients and services generated for Dart, Swift, Java, and TypeScript automatically.

Multiple Views (Web)

Web platforms support multiple Flutter views within the same application, each with independent communication channels.

Start Parameters

Pass structured initialization data when launching Flutter. Define your StartParams message in proto — include whatever your app needs.

Ready-to-Ship Packages

Generates CocoaPods for iOS, AAR for Android, and npm packages for web — ready for distribution to your teams.

Get Started in 3 Steps

From Flutter app to embeddable module in minutes

1

Configure

Add flutter_embedding to your pubspec.yaml and define your proto handovers for type-safe communication.

2

Generate

Run a single CLI command to generate modules for your target platforms with example apps.

3

Integrate

Import the generated package into your host application and start communicating with Flutter.

Terminal 
# Install the CLI
dart pub add dev:flutter_embedding_cli

# Generate iOS module with example app
dart run flutter_embedding_cli:generate ios --example

# Generate Android module with example app
dart run flutter_embedding_cli:generate android --example

# Generate React Native module with example app
dart run flutter_embedding_cli:generate react-native --example

# Generate React web module with example app
dart run flutter_embedding_cli:generate web-react --example

# Generate Angular web module with example app
dart run flutter_embedding_cli:generate web-angular --example
handovers_to_flutter.proto 
// Define what the host can call on Flutter
service HandoversToFlutter {
  rpc SetTheme(ThemeRequest) returns (Empty);
  rpc UpdateUser(UserData) returns (Empty);
  rpc NavigateTo(Route) returns (Empty);
}

message StartParams {
  string environment = 1;
  string auth_token = 2;
  UserData user = 3;
}
handovers_to_host.proto 
// Define what Flutter can call on the host
service HandoversToHost {
  rpc Exit(Empty) returns (Empty);
  rpc ShareContent(ShareRequest) returns (Empty);
  rpc GetDeviceInfo(Empty) returns (DeviceInfo);
  rpc OpenPayment(PaymentRequest) returns (PaymentResult);
}

// Define any message types you need
message PaymentRequest {
  string product_id = 1;
  int32 amount = 2;
}

Define your services and messages in proto files — the CLI generates type-safe code for all platforms automatically.
